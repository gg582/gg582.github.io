이번 작업은 OpenGL 없이 Utah teapot을 X11 창에 직접 그리는 소프트웨어 렌더러를 만드는 일이었다. 먼저 기본적인 조명, 곡률 보정, 메시 접합 알고리즘의 성질과 부작용을 검토했다. 그 뒤 AI 코더에게 `teapot.bpt`의 32개 Bezier 패치를 삼각형으로 쪼개고, CPU에서 z-buffer와 Blinn-Phong을 계산한 뒤 `XPutImage`로 프레임버퍼를 넘기는 파이프라인을 단계적으로 구축하게 했다.

처음 지시는 법선에 난수를 섞고 bilateral 필터로 이를 정리하는 방향이었다. 결과는 좋지 않았다. 필터가 해결해야 할 문제를 입력 단계에서 먼저 만들고 있었기 때문이다. 이후 AI 코더에게 난수 제거, 곡률 스파이크만의 제한적 보정, 점광원 계산, 전역 정점 용접 순으로 수정하게 했다. 난수를 없앤 뒤에도 몸통에는 십자 접합선이 남았고, 원인은 필터 강도가 아니라 같은 위치의 패치 경계 정점을 서로 다른 정점으로 보관한 메시 구조였다. 이 글은 그 진단과 수정 과정을 PNG 결과와 실제 코드 위치를 따라 적는다.

소스와 입력 데이터는 [utah-teapot-software-renderer.zip](/assets/files/utah-teapot-software-renderer.zip)에 넣었다. 압축 파일에는 `main.cpp`, `CMakeLists.txt`, `teapot.bpt`가 들어 있다.

## 1. 처음 보인 것은 렌더링 경로였다

![초기 X11 실시간 미리보기](/assets/images/utah-teapot/teapot_realtime_preview.png)

`teapot_realtime_preview.png`는 최종 음영 비교용 이미지는 아니다. 기존 Batch4D 경로가 패치 표본과 접선 방향을 점과 선으로 표시한 화면이다. 이 이미지에서 확인한 것은 두 가지다. 패치 데이터는 읽히고 있었고, X11 창으로 CPU 프레임버퍼를 내보내는 경로도 이미 동작하고 있었다. 따라서 이후의 문제는 창 시스템이 아니라 메시와 음영 단계에 있었다.

## 2. 난수를 제거하고 raw normal과 bilateral normal을 비교했다

| Bilateral OFF | Bilateral ON |
| --- | --- |
| ![clean off](/assets/images/utah-teapot/clean_off.png) | ![clean on](/assets/images/utah-teapot/clean_on.png) |

왼쪽 `clean_off.png`는 Bezier 표본에서 계산한 raw normal만 쓴 결과다. 오른쪽 `clean_on.png`는 거리와 법선 방향 차이로 이웃 법선을 가중한 결과다. 두 그림의 차이가 크지 않은 것은 실패가 아니다. 임의 노이즈를 제거했으므로 bilateral은 원래 표면을 새로 만드는 단계가 아니라 작은 법선 불안정성만 제한적으로 보정해야 한다.

그러나 두 그림 모두 몸통의 세로 음영 분할을 완전히 없애지 못한다. 이 시점에는 패치 경계가 서로 다른 정점 집합이기 때문이다. 법선만 평균내는 방식으로는 각 패치의 이웃 그래프가 끊긴 문제를 고칠 수 없다.

## 3. 분할 수를 늘려도 접합선은 남는다

| 패치당 32 분할, Bilateral OFF | 패치당 32 분할, Bilateral ON |
| --- | --- |
| ![smooth32 off](/assets/images/utah-teapot/smooth32_off.png) | ![smooth32 on](/assets/images/utah-teapot/smooth32.png) |

`smooth32_off.png`와 `smooth32.png`는 patch당 tessellation을 32로 올린 시험이다. 삼각형 내부의 거친 변화는 줄었지만, 몸통 패치가 만나는 세로 경계는 여전히 보인다. 즉 더 많은 삼각형은 표본 밀도를 올릴 뿐, 분리된 패치를 통짜 표면으로 만들지 않는다.

여기서 결론은 간단했다. 필터를 더 세게 거는 대신, 먼저 계산 대상 자체를 고쳐야 한다.

## 4. 광원도 거리와 곡률을 보게 했다

| 초기 점광원 값 | 조정한 점광원 값 |
| --- | --- |
| ![physical check](/assets/images/utah-teapot/physical_check.png) | ![physical final](/assets/images/utah-teapot/physical_final.png) |

점광원은 모든 표면에 같은 세기를 주는 방향광보다 설명력이 좋다. 각 픽셀에서 광원까지 거리를 구하고 역제곱 감쇠와 간단한 Beer-Lambert 투과를 곱했다. 왼쪽 `physical_check.png`는 광원 세기를 과하게 둔 중간 결과다. 몸통과 주둥이의 하이라이트가 포화되어 표면의 형태를 읽기 어렵다. 오른쪽 `physical_final.png`에서는 광원 세기를 낮춰, 반사광의 위치와 몸통의 밝기 구배가 다시 분리된다.

`main.cpp`의 1048-1065행이 그 계산이다.

```cpp
Vec3 toLight = sp.lightPos - P;
const double distance2 = std::max(0.04, dot(toLight, toLight));
const double distance = std::sqrt(distance2);
Vec3 L = toLight / distance;
const double transmission = std::exp(-sp.airExtinction * distance);
const double radiance = sp.lightIntensity * transmission / distance2;

const double roughness = std::clamp(
    sp.baseRoughness + sp.curvatureRoughness * curvature, 0.08, 0.85);
const double effectiveShininess = std::max(2.0,
    sp.shininess / (1.0 + 5.0 * roughness * roughness));
```

여기서 `curvature`는 물리적으로 측정한 마찰 계수가 아니다. 이웃 중심으로부터의 거리로 만든 기하학적 곡률 근사다. 곡률이 낮은 넓은 구간은 하이라이트를 더 날카롭게 유지하고, 변화가 큰 구간은 하이라이트를 조금 넓힌다.

## 5. EWMA는 전체를 깎는 필터가 아니다

곡률 보정은 `main.cpp` 851-906행에 있다. 먼저 가까운 이웃의 가우시안 가중 중심을 구해 곡률 근사를 만들고, 다시 이웃 곡률과 현재 곡률을 섞어 EWMA 추세를 만든다.

```cpp
const double neighbourTrend = weighted / weight;
ewma[v] = alpha * curvature[v] + (1.0 - alpha) * neighbourTrend;

const double excess = curvature[v] - ewma[v] * 1.18;
if (excess <= 0.0) continue;
const double amount = std::clamp(excess * 0.025, 0.0, 0.035);
updated[v] = m.pos[v] * (1.0 - amount) + centroid[v] * amount;
```

중요한 것은 마지막 세 줄이다. 모든 정점을 이웃 평균으로 옮기지 않는다. 현재 곡률이 주변 추세보다 충분히 큰 경우에만, 최대 3.5%만 중심 쪽으로 이동시킨다. 그래서 주둥이, 뚜껑, 림처럼 원래 곡률이 있는 부분을 통째로 뭉개는 대신, 한 정점만 유난히 튀는 경우를 겨냥한다.

파이프라인의 순서를 정할 때에는 구장산술과 구일집 계열 문제를 반복해 풀며 익힌 계산 습관을 기준으로 삼았다. 큰 대상을 바로 만지기보다 계산 가능한 단위로 나누고, 중간값을 쌓고, 오차가 생긴 위치에만 다음 보정을 적용하는 방식이다. 이를 바탕으로 AI 코더에게 전체를 평균내지 말고 국소 곡률 추세를 먼저 계산한 뒤 스파이크만 제한적으로 움직이도록 지시했다. 현대의 EWMA나 bilateral 필터가 고전 산학에서 나온 것은 아니다. 다만 이들을 조합할 때 어떤 값을 먼저 만들고 어디를 다시 계산할지 결정하는 데 그 훈련이 직접 작용했다.

## 6. 십자 접합선의 원인은 필터가 아니라 정점 소유권이었다

| 패치별 정점 유지, `--unify=no` | 전역 정점 용접, `--unify=yes` |
| --- | --- |
| ![split mesh](/assets/images/utah-teapot/split.png) | ![unified mesh](/assets/images/utah-teapot/unified.png) |

왼쪽 `split.png`에서는 몸통 사분면이 같은 위치를 공유해도 경계 정점이 별도 인덱스다. 각 정점의 이웃 목록과 누적 법선이 경계를 넘지 못하므로, 정면 하이라이트 부근에 세로선 또는 십자선이 남는다.

오른쪽 `unified.png`는 `--unify=yes`를 준 결과다. 경계 정점을 하나로 용접한 뒤, 곡률과 법선을 그 전역 그래프에서 다시 계산했다. 이 단계가 접합선을 줄이는 핵심이다. tessellation을 32로 늘린 것보다 메시 연결성을 바로잡는 편이 더 직접적으로 작동한다.

실제 용접은 `main.cpp` 750-836행에서 수행한다. 755행의 `weldTolerance`는 같은 정점으로 취급할 거리 허용치이고, 761-785행의 `addVertex`는 이미 가까운 전역 정점이 있으면 새 정점을 만들지 않고 기존 인덱스를 반환한다.

```cpp
constexpr double weldTolerance = 2e-4;

if (it != weldedVertex.end() &&
    length(m.pos[it->second] - p) <= weldTolerance)
    return it->second;
```

용접이 끝난 뒤에도 할 일이 남는다. 813-836행은 각 패치에서 얻은 이웃 관계를 같은 전역 정점의 adjacency에 누적하고 CSR 형식으로 평탄화한다. 이 구조를 사용하므로 곡률 EWMA와 bilateral 필터가 경계를 건너 이웃을 볼 수 있다.

```cpp
connect(grid[idx(i, j)], grid[idx(ni, nj)]);

std::sort(neighbours.begin(), neighbours.end());
neighbours.erase(std::unique(neighbours.begin(), neighbours.end()), neighbours.end());
m.nbrList.insert(m.nbrList.end(), neighbours.begin(), neighbours.end());
```

## 7. 실행

```bash
cmake -S . -B build
cmake --build build -j
cd build
./teapot --unify=yes
```

X11 없이 이미지 파일만 만들려면 다음처럼 실행한다.

```bash
./teapot --still teapot.ppm --unify=yes
```

기본 제어는 마우스 드래그로 orbit, 휠로 zoom, `b`로 bilateral 토글, `1`과 `2`로 spatial sigma 조정, `3`과 `4`로 range sigma 조정, `5`와 `6`으로 shininess 조정이다.

이 작업에서 남은 결론은 단순하다. 표면이 찌그러져 보인다고 해서 바로 더 강한 스무딩을 넣을 일은 아니다. 먼저 그 불연속이 표본 밀도 문제인지, 법선 문제인지, 아니면 동일한 경계 정점을 서로 다른 객체로 보관한 연결성 문제인지 분리해야 한다. 이번 티팟에서는 마지막 경우가 가장 컸다.
